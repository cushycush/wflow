# Flatpak — wflow on Flathub

This directory holds the Flatpak manifest, AppStream metadata, desktop
entry, and build helpers for shipping wflow on Flathub.

## Spike findings (2026-04-25)

The v0.3 plan named this a P0 spike because the load-bearing question
is whether wflow can actually execute shell commands inside a Flatpak
sandbox. **Answer: yes, with caveats.**

### The strategy that works

`flatpak-spawn --host -- <command>` runs `<command>` on the host
machine, with the host's PATH, environment, and full filesystem. The
sandbox is allowed to invoke this when the manifest grants
`--talk-name=org.freedesktop.Flatpak`, which gives access to the
specific D-Bus interface the helper uses (it does **not** open the
whole session bus).

wflow's runtime detection: presence of the `FLATPAK_ID` env var means
we're sandboxed, and every host-bound command goes through the
`crate::host::host_command` helper, which prepends `flatpak-spawn
--host --` to the argv. Outside a sandbox, `host_command` is a plain
`Command::new` — no per-call branching at the call site.

Real-world precedents that ship on Flathub today using this exact
pattern:

- **Visual Studio Code** (`com.visualstudio.code`)
- **GNOME Builder** (`org.gnome.Builder`)
- **GNOME Boxes** (`org.gnome.Boxes`)

### The caveats

Flathub's review team treats `--talk-name=org.freedesktop.Flatpak` as
"wide" permission and flags it for extra scrutiny. The submission will
not auto-reject, but expect:

- Reviewer questions about *why* the app needs host execution. The
  honest answer: workflows *are* the host commands; an automation
  tool that can't drive the user's actual desktop session is a
  workflow viewer, not a workflow runner. `REVIEW.md` at the repo
  root documents the trust model the user opts into when running
  unfamiliar workflows.
- A nudge toward narrower permissions where possible (we already
  removed `--talk-name=org.freedesktop.portal.Notification` because
  `notify-send` runs on the host, not via portal).
- Slower review turnaround than the average Flathub submission —
  budget weeks, not days.

If the review goes badly, the fallback is documented in the v0.3 plan:
demote Flathub to a stretch goal, ship v0.3 to AUR + cargo-install
only, retry Flathub for v0.4.

### What's NOT verified yet

This whole spike was authored on a machine without `flatpak-builder`
installed. The following needs final verification on a Flatpak-enabled
machine before submitting:

1. **Build**: `./packaging/flatpak/build-local.sh` runs without error
   and produces a working `flatpak run io.github.cushycush.wflow`.
2. **GUI launches**: Qt 6.7 from KDE Platform 6.7 starts cleanly under
   Wayland and XWayland fallback.
3. **Host-spawn round trip**: a workflow whose only step is
   `shell "id > /tmp/wflow-flatpak-test"` succeeds, and the file appears
   *on the host* at `/tmp/wflow-flatpak-test` with the user's real
   uid (not the sandbox uid).
4. **cxx-qt at runtime**: the GUI's QML modules load (no `Cannot find
   QML module` errors from the sandboxed Qt resolver).
5. **Record Mode portal**: ashpd's RemoteDesktop request opens the
   host's xdg-desktop-portal dialog (sandboxed apps reach the portal
   automatically; no extra `--talk-name` needed).

If any of those fail, the manifest needs another iteration — start by
reading `flatpak-builder` build logs and Flathub's quick-launcher
checks (`flatpak-builder-checks.sh` from
`flathub/flatpak-external-data-checker`).

## Building locally

Install the runtimes (one-time setup):

```sh
flatpak remote-add --if-not-exists --user flathub \
    https://flathub.org/repo/flathub.flatpakrepo
flatpak install --user flathub \
    org.kde.Platform//6.7 org.kde.Sdk//6.7 \
    org.freedesktop.Sdk.Extension.rust-stable//24.08
```

Then from the repo root:

```sh
./packaging/flatpak/build-local.sh
flatpak run io.github.cushycush.wflow
```

The script regenerates `cargo-sources.json` against the current
`Cargo.lock` automatically.

## Submitting to Flathub

Flathub apps live as one repository per app at
`github.com/flathub/<app-id>`. The submission process:

1. Tag a release in this repo (e.g. `v0.3.0`).
2. Switch the manifest's `sources` to a `git`+`tag` entry pointing at
   the tag — the dir-based source in this file is for local testing
   only.
3. Open a PR against `github.com/flathub/flathub` adding
   `io.github.cushycush.wflow` to `flathub.json`.
4. After review, Flathub creates `github.com/flathub/io.github.cushycush.wflow`
   for ongoing maintenance — every new release is a PR there.

Per-version release upkeep: bump `<release>` in `metainfo.xml`, regenerate
`cargo-sources.json` from the new `Cargo.lock`, point the `git` source at
the new tag, open a PR to the per-app Flathub repo.
