# Flatpak — wflow on Flathub

This directory holds the Flatpak manifest, AppStream metadata, desktop
entry, and build helpers for shipping wflow on Flathub.

## Spike findings (2026-04-25)

The v0.3 plan named this a P0 spike because the load-bearing question
is whether wflow can actually execute shell commands inside a Flatpak
sandbox. **Answer: yes, with caveats.**

### The strategy that works

`flatpak-spawn --host -- <command>` runs `<command>` on the host
machine, with the host's PATH, environment, and full filesystem. The
sandbox is allowed to invoke this when the manifest grants
`--talk-name=org.freedesktop.Flatpak`, which gives access to the
specific D-Bus interface the helper uses (it does **not** open the
whole session bus).

wflow's runtime detection: presence of the `FLATPAK_ID` env var means
we're sandboxed, and every host-bound command goes through the
`crate::host::host_command` helper, which prepends `flatpak-spawn
--host --` to the argv. Outside a sandbox, `host_command` is a plain
`Command::new` — no per-call branching at the call site.

Real-world precedents that ship on Flathub today using this exact
pattern:

- **Visual Studio Code** (`com.visualstudio.code`)
- **GNOME Builder** (`org.gnome.Builder`)
- **GNOME Boxes** (`org.gnome.Boxes`)

### The caveats

Flathub's review team treats `--talk-name=org.freedesktop.Flatpak` as
"wide" permission and flags it for extra scrutiny. The submission will
not auto-reject, but expect:

- Reviewer questions about *why* the app needs host execution. The
  honest answer: workflows *are* the host commands; an automation
  tool that can't drive the user's actual desktop session is a
  workflow viewer, not a workflow runner. `REVIEW.md` at the repo
  root documents the trust model the user opts into when running
  unfamiliar workflows.
- A nudge toward narrower permissions where possible (we already
  removed `--talk-name=org.freedesktop.portal.Notification` because
  `notify-send` runs on the host, not via portal).
- Slower review turnaround than the average Flathub submission —
  budget weeks, not days.

If the review goes badly, the fallback is documented in the v0.3 plan:
demote Flathub to a stretch goal, ship v0.3 to AUR + cargo-install
only, retry Flathub for v0.4.

### Verification status (2026-04-25)

Verified end-to-end on a real Arch host. Four of five items confirmed,
one user-interactive item pending:

| # | Item | Status |
|---|---|---|
| 1 | `build-local.sh` produces a working install | ✅ |
| 2 | Qt 6.9 GUI launches under Wayland | ✅ (clean launch, zero stderr) |
| 3 | Host-spawn round trip writes to host fs with real uid | ✅ uid=1000 on host, FLATPAK_ID unset in spawned shell |
| 4 | cxx-qt's QML modules resolve at runtime | ✅ (no QML errors in launch log) |
| 5 | ashpd RemoteDesktop portal opens host dialog | ⏳ click `● Record` and confirm portal prompt |

Three real issues surfaced during verification and have been fixed in
the manifest / build script. If verification ever needs to repeat:

- The `flatpak-cargo-generator.py` Python deps (`aiohttp`, `tomlkit`)
  live in a self-contained venv at `target/cargo-gen-venv/` that
  `build-local.sh` creates automatically.
- The **`25.08` SDK extension is required** (Rust 1.95+). Older
  `24.08` ships Rust 1.81 which lacks edition2024 — required by
  ashpd's transitive deps via zbus → zvariant_utils.
- An icon at `io.github.cushycush.wflow.svg` is installed to
  `/app/share/icons/hicolor/scalable/apps/`. AppStream validation
  fails without it.

One non-blocking caveat worth knowing:

- The library is **shared** between Flatpak and host installs —
  Flatpak's `--filesystem=xdg-config/wflow:create` bind-mounts the
  host's `~/.config/wflow` into the sandbox at both `~/.config/wflow`
  AND `~/.var/app/<id>/config/wflow`. Same files either way. *Good*
  news for users who switch install methods. `wflow path` reports
  the per-app alias path, which is technically correct (it resolves
  to the same files) but cosmetically misleading.

## Building locally

Install the runtimes (one-time setup):

```sh
flatpak remote-add --if-not-exists --user flathub \
    https://flathub.org/repo/flathub.flatpakrepo
flatpak install --user flathub \
    org.kde.Platform//6.9 org.kde.Sdk//6.9 \
    org.freedesktop.Sdk.Extension.rust-stable//25.08
```

Then from the repo root:

```sh
./packaging/flatpak/build-local.sh
flatpak run io.github.cushycush.wflow
```

The script regenerates `cargo-sources.json` against the current
`Cargo.lock` automatically.

## Submitting to Flathub

There are two manifests in this directory:

- `io.github.cushycush.wflow.yaml` — local-dev variant. Uses a `dir`
  source so `build-local.sh` can build the working tree.
- `io.github.cushycush.wflow.flathub.yaml` — submission variant. Uses
  a `git`+`tag` source so reviewers and end-users build from a
  verifiable revision. **This is the file that gets copied into the
  Flathub per-app repo.**

Flathub apps live as one repository per app at
`github.com/flathub/<app-id>`. The submission process:

1. Tag a release in this repo (e.g. `v0.3.0`) and verify the
   `tag` field in `io.github.cushycush.wflow.flathub.yaml` matches.
   Optionally pin the `commit` SHA below the tag for reproducibility
   (`git rev-parse v0.3.0`).
2. Bump `<release>` in `metainfo.xml` if cutting a new version, and
   regenerate `cargo-sources.json` from the current `Cargo.lock`
   (`build-local.sh` does this automatically; otherwise run
   `flatpak-cargo-generator.py Cargo.lock -o packaging/flatpak/cargo-sources.json`).
3. Open a PR against `github.com/flathub/flathub` adding
   `io.github.cushycush.wflow` to `flathub.json`. Submit the
   `*.flathub.yaml` manifest plus the metainfo, desktop, and icon
   files as the seed for the per-app repo.
4. After review, Flathub creates
   `github.com/flathub/io.github.cushycush.wflow` for ongoing
   maintenance — every new release is a PR there with the same set
   of files.

Per-version release upkeep: bump `<release>` in `metainfo.xml`,
regenerate `cargo-sources.json`, bump the `tag` in
`*.flathub.yaml`, open a PR to the per-app Flathub repo.
