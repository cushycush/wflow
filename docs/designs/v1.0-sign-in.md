# v1.0 sign-in: browser handoff + favorites tab

The last v1.0 gate. Desktop authenticates against wflows.io, reads
`/me`, and shows a "My favorites" tab populated from the user's
account. Same path enables comments and (eventually) publishing
from the desktop.

## State of the world

- Better Auth is mounted at `/api/auth` on `wflows.vercel.app`. `get-session` returns the canonical `null` for unauthed and a session JSON when a cookie is set.
- Better Auth is browser-oriented. There's no first-party "device flow" or "personal access token" endpoint today.
- The desktop has no auth surface at all — no UI, no token storage, no authenticated bridge calls.
- The deeplink import path (`wflow://import?source=...`) is already wired through `xdg-open`. We can reuse the same handoff for sign-in.

## Recommended flow: browser sign-in + deeplink token return

Reuses the existing `wflow://` deeplink wire — no manual paste, no polling, no embedded webview. Best UX on a Linux desktop.

```
1. User clicks "Sign in" in the desktop.
2. Desktop generates a single-use nonce.
3. Desktop opens https://wflows.io/auth/desktop?nonce=<nonce> via xdg-open.
4. Browser hands the URL to the user's default browser.
5. User signs in (magic link / GitHub / Discord OAuth) on the web.
6. After sign-in, the page redirects to wflow://auth/callback?nonce=<nonce>&token=<token>
7. xdg-open hands the URL back to the desktop's existing wflow:// scheme handler.
8. Desktop verifies nonce matches, persists token, fetches /api/v0/me.
9. Desktop UI flips to "Signed in as @handle".
```

The nonce closes a single-window race: a malicious page can't fire `wflow://auth/callback?token=ATTACKER` at the desktop because the desktop only accepts a callback whose nonce matches the one it generated for this sign-in.

### Why not OAuth Device Code Flow

RFC 8628 (the GitHub CLI / streaming-stick pattern: "go to https://x and enter code ABC1-XYZ2") is the textbook answer for headless device sign-in. It's a worse fit here because:

- We're not headless — the desktop has a browser one xdg-open away.
- Device flow needs a polling loop; the deeplink path is single-shot.
- Manual code entry is friction that the OS-native deeplink wire avoids.

We can fall back to a polling-style flow as a second option for users whose `xdg-open` is broken (rare on KDE/GNOME/Hyprland/Sway), but the deeplink is the primary.

### Why not embedded webview

Doubles the binary size, gives the auth provider an unfamiliar Chromium build, and forces the user to type their password into a window that doesn't look like Firefox. Browser handoff is the answer everywhere modern desktop apps land (1Password, Linear, Notion, ...).

## Server-side requirements (wflows.io)

Three endpoints, all on `wflows.vercel.app/api/v0/auth/desktop/*`. Need to be built before the desktop side ships.

### `GET /auth/desktop?nonce=<nonce>`

User-facing page. Renders the existing Better Auth sign-in surface (magic link / GitHub / Discord), with the `nonce` carried through every form submit and OAuth round-trip via state parameter. On successful sign-in, redirect to `wflow://auth/callback?nonce=<nonce>&token=<personal-access-token>`.

Token shape: opaque random string, persisted server-side in a `desktop_tokens` table tied to a `user_id`, scoped to read-only by default. UI for revoking lives at `/settings/devices` on the web.

### `POST /api/v0/auth/desktop/exchange`

Body: `{ nonce: string }`. Returns: `{ token: string, user: { handle, displayName, avatar } }` — only if the nonce was issued in the last 5 minutes and hasn't already been redeemed. Used by the polling fallback flow when deeplink is unavailable; not strictly required for the primary deeplink flow but cheap to add and lets us log nonce reuse attempts.

### `GET /api/v0/me`

Authenticated. Bearer token in the `Authorization` header. Returns: `{ handle, displayName, avatar, favorites: [...], createdAt }`. The desktop hits this on startup if a token is stored, on every "Sign in" success, and to refresh the profile / favorites tab.

### `GET /api/v0/favorites`

Authenticated. Returns the user's favorited workflows in the same shape as `/api/v0/browse`. Probably just an alias / scoped query rather than a fresh schema.

## Desktop-side surface

### Settings page: Account section

A new section in `qml/pages/SettingsPage.qml`, above the existing Theme / Motion blocks:

- **Signed out state:** `[ Sign in with wflows.io ]` button. Click → calls `AuthController.start_sign_in()`.
- **Signed out, in-flight state:** spinner + `Waiting for browser sign-in… (cancel)`. Click cancel → drops the pending nonce, returns to signed-out state.
- **Signed in state:** avatar + handle + "Signed in" + `[ Sign out ]` button. Click sign-out → drops the local token + opens browser to revoke server-side.
- **Sign-in failed state:** `Couldn't complete sign-in` + reason + retry button.

### Bridge: AuthController

New cxx-qt bridge at `src/bridge/auth.rs`. Mirrors ExploreController's shape (async invokables, Qt-thread queue for results).

```rust
#[qproperty(QString, state)]      // "signed_out" | "pending" | "signed_in" | "failed"
#[qproperty(QString, handle)]     // empty when signed_out
#[qproperty(QString, display_name)]
#[qproperty(QString, avatar_url)]
#[qproperty(QString, last_error)]

#[qinvokable] fn start_sign_in(self);   // generates nonce, opens browser
#[qinvokable] fn cancel_sign_in(self);  // drops pending nonce
#[qinvokable] fn complete_sign_in(self, token: QString);  // called by deeplink handler
#[qinvokable] fn sign_out(self);

#[qsignal]    fn sign_in_succeeded(self, handle: QString);
#[qsignal]    fn sign_in_failed(self, reason: QString);
```

### Deeplink scheme expansion

The existing `wflow://import?source=...` handler in `qml/Main.qml` parses one shape. Extend `_resolveDeeplink` to recognize `wflow://auth/callback?nonce=...&token=...` and route to `authController.complete_sign_in(token)` when the nonce matches the stored pending nonce.

The desktop registers `x-scheme-handler/wflow` already (per the .desktop entry); no packaging change.

### Token persistence

Add a `desktop_token` field to `state.toml` via the existing `StateController`. Storing in plaintext is the same trust model as the workflow library being on disk — the user's own home directory. We can revisit with libsecret integration as a follow-up if anyone asks.

### Authenticated bridge calls

`ExploreController` and any new controller that needs the user's identity reads the token from `StateController` and adds an `Authorization: Bearer <token>` header to its `reqwest` calls. On 401 from any authenticated call, the controller emits `auth_expired` so the UI can flip back to the signed-out state.

## Favorites tab

Once auth lands, add a `favorites` page mode in `qml/Main.qml`. Renders the existing `CommunityCard` grid (the v0 `/api/v0/favorites` response shape matches `/api/v0/browse`), with a different empty state ("Star a workflow on Explore to find it here"). Hides entirely when signed-out — no point teasing the tab to anonymous users.

## What ships in v1.0

Sign-in flow + favorites tab. That closes the v1.0 gate the BACKLOG names. Posting comments and publishing from the desktop are post-1.0 — they need their own design rounds.

## Implementation order

1. **Server side** (separate repo): build the three endpoints, deploy to wflows.vercel.app.
2. **Desktop scaffolding**: AuthController bridge, Settings → Account section, deeplink handler extension. Can ship with a hardcoded staging origin while the server side lands.
3. **Polish**: favorites tab, 401 handling on existing calls, sign-out flow, error states.
4. **QA**: KDE / GNOME / Hyprland / Sway, verify the deeplink round-trip on each.

The server work is the gating item. Until that lands, desktop scaffolding can build against a mock token issuance endpoint or just skip directly to a hand-pasted token for development.
