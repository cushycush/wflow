# wflow v0.4 — leg-1 wake up: trigger-mode daemon

Source: planning conversation on 2026-04-25. Builds on the v0.3
leg-2 plan. Status: spec, not a commitment.

## The problem

AutoHotkey is the gravitational center of "I want to automate my
desktop on a keyboard shortcut." On Linux that audience has Hawck
and keymapper for raw remapping, but nothing that gives them
AutoHotkey's actual headline experience: bind a hotkey to a *script*
that does whatever you want. wflow has the script half (the engine,
the KDL language, the GUI editor) but no way to fire a workflow
without launching the GUI or running `wflow run` in a terminal. As
long as that's true, the AHK audience never reaches the parts of
wflow that are good.

The wdotool-core migration in v0.3.1 made this cheaper than it
looked at v0.3 planning time. The engine is in process, the libei
portal handshake is reusable, and ashpd 0.13 already exposes
`GlobalShortcuts`. The bones of a daemon are already in the binary;
v0.4's job is to grow a triggering surface around them.

## What ships in v0.4

A `wflow daemon` subcommand that registers global hotkeys via
`org.freedesktop.portal.GlobalShortcuts` (with a wlroots-IPC
fallback for Sway and Hyprland), and runs the bound workflow when
the shortcut fires. Hotkey bindings live in a new
`~/.config/wflow/triggers.kdl` file, hand-authorable in the same
spirit as the workflow files themselves. The GUI gains a Triggers
tab where bindings can be edited; the daemon picks up changes
without restart.

Out of scope for v0.4: hotstrings (text expansion), per-window
triggers, file-watch triggers, schedule (cron) triggers, and
multi-user system-level installs. Each of those is a real product
in its own right; doing them at the same time would dilute the
launch.

## How it works

### Hotkey schema

`triggers.kdl` is a flat list of bindings. Each binding has an id,
a chord, the workflow it runs, and optional metadata for the
GlobalShortcuts portal (description, preferred-trigger).

```kdl
schema 1

trigger "open-terminal" {
    chord "ctrl+alt+t"
    workflow "dev-setup"
    description "Open kitty + Firefox tiled"
}

trigger "morning" {
    chord "super+shift+m"
    workflow "morning-standup"
}
```

The chord syntax is the same one wflow already uses inside `key`
actions, so users who've hand-authored a workflow already know how
to write a binding. `workflow` is a library id, the same one
`wflow run` accepts.

### Daemon process model

Single instance per user, locked via an abstract Unix socket at
`@wflow-daemon-$UID`. Starting a second `wflow daemon` is a no-op
that prints "already running, pid N" and exits zero. The GUI launches
the daemon on first open if it isn't running and notes the pid.

The daemon owns one tokio runtime, one libei backend (lazy,
inherited from the in-process engine), and one D-Bus connection.
On startup it reads `triggers.kdl`, registers the shortcuts via
`GlobalShortcuts.bind_shortcuts`, and then `Activated` signals come
back as a stream the daemon dispatches to `engine::run_workflow`.
File-watching `triggers.kdl` for hot reload uses `notify` (small
crate, already common; a tokio FS watcher would also work).

### IPC contract

Daemon exposes a D-Bus interface at `org.cushycush.wflow.Daemon`
with three methods to start. `Reload()` re-reads `triggers.kdl`.
`ListBindings() -> Vec<Binding>` enumerates active shortcuts so the
GUI can render the Triggers tab. `RunWorkflow(id: String)` lets the
GUI fire a workflow through the same path a hotkey would, so manual
GUI runs and hotkey-driven runs share state and history.

D-Bus over the session bus, not the system bus. No setuid, no
elevated capabilities. The portal does the privileged work
(registering the global shortcut with the compositor); the daemon
itself stays an ordinary user-session process.

### Compositor backends

Two-layer dispatch behind a `TriggerBackend` trait, shaped the same
way `wdotool-core::Backend` is. The portal backend tries
`GlobalShortcuts.bind_shortcuts` first; on KDE 6 and GNOME 46+ this
is the right answer. If the portal call fails (older xdp, sway,
hyprland), fall through to a compositor-IPC backend that talks
hyprctl / swaymsg to register a binding that runs `wflow run <id>
--yes`. Slower (subprocess per fire) but works on the wlroots side
of the desktop where users live in a config file anyway.

Single backend trait, two implementations, the same lazy-init story
as the input backend. Detection mirrors `wdotool-core::detector`
(check `XDG_CURRENT_DESKTOP`, `HYPRLAND_INSTANCE_SIGNATURE`,
`SWAYSOCK`).

## What's deferred and why

**Hotstrings (`btw → by the way`).** Needs a global keyboard
*monitor*, not just a binder. We have the libei stack for input
emission and we have the receiver path from Record Mode, but
combining them for low-latency monitoring without battery cost is
its own project. Defer to v0.5 with the rest of the trigger
expansion work.

**Per-window triggers.** "When Firefox is foreground, ctrl+t opens
my reading list, otherwise it does the default." Needs both the
trigger daemon and a window-state monitor that's cheap enough to
run continuously. The daemon already does the second half through
the wdotool-core window APIs, but the binding model gets thornier
(per-binding window predicate, fallback chains). Worth a v0.5 plan
of its own.

**File-watch and schedule triggers.** Cron-style and inotify-style
triggers fall out cheaply once the daemon exists, but they're a
different product from "AHK on Linux." Treat them as v0.5 surface.

## Risks

The biggest is that the GlobalShortcuts portal isn't as widely
deployed as the v0.3 plan assumed. Plasma 6 and GNOME 46 ship it,
but on older Plasma 5 boxes and on minimal compositors the portal
call fails. Mitigation is the wlroots-IPC fallback above; ugly but
keeps the audience reachable.

Second risk is the AHK migration cost. AHK users have years of
muscle memory in `^!p::Send, hello`. wflow's KDL syntax doesn't
look anything like that. Counter is the migration page in docs,
side by side, and a `wflow import-ahk` subcommand later that makes
a best effort on simple Send-and-Run scripts. Don't ship the
importer in v0.4; ship the migration page.

Third risk is scope drift. Once the daemon exists, every "wouldn't
it be cool if" trigger feels lossless to add. Defer hotstrings,
per-window, file-watch, and schedule triggers to v0.5 explicitly.
v0.4 ships hotkey-fires-workflow and stops.

## What's left (post-v0.4)

If v0.4 lands and the audience metric (distinct GitHub issue
authors who name AHK in their use case) trends up, v0.5 is the
trigger-expansion release: hotstrings, per-window predicates,
schedule, file-watch. If it doesn't trend up, the leg-1 commitment
gets re-evaluated rather than doubled down on. The dormancy
declaration in the v0.3 plan was load-bearing; staying honest about
which leg is pulling its weight is worth more than any one
release.
