# wflow 1.0.1: triggers do what they say

Two patches to the trigger surface that 1.0 had ignored.

## Trigger when-predicates actually gate now

The GUI's chord-capture dialog has been letting you author
`when window-class=firefox` predicates since 1.0, but the daemon
ignored them. Pressing `super+t` with `when window-class=firefox`
fired the workflow no matter what app was focused. Silent
misbehavior on a feature the UI advertises.

Daemon binds (Hyprland, Sway, GlobalShortcuts portal) now exec a
new internal `wflow trigger-fire <id>` wrapper instead of
`wflow run`. The wrapper loads the workflow, reads `trigger.when`,
probes the focused window, and only then fires. On Hyprland it
reads `hyprctl activewindow -j` over IPC; on Sway it walks the i3
tree for the focused leaf. KDE Plasma 6 and GNOME 46+ portal users
fail open (chord still fires, no gating, debug-trace line in
`journalctl --user -u wflow-daemon`) until the per-DE active-window
probes land.

Predicate semantics:

- No `when` clause: fire as before.
- `window-class="x"`: fire iff the focused window's class contains
  `x`, case-insensitive.
- `window-title="x"`: fire iff the focused window's title contains
  `x`, case-insensitive.

A skipped fire is silent (exit 0, single info-level trace line) so
the chord registers as a no-op rather than a noisy failure on every
press in the wrong window. `wflow run` stays the ungated public
path for cron and scripts; the gate is daemon-context only via the
hidden subcommand.

## Imported workflows surface their chords before binding

Workflows on wflows.io can carry `trigger { chord ... }` blocks.
The import path already preserved them through the KDL roundtrip,
so the chord registered globally as soon as the daemon's file
watcher picked the new file up. Problem: you never saw the chord
coming. You clicked Open in wflow, hit Install, and a moment later
`super+t` fired the imported workflow instead of whatever you had
before.

The deeplink confirm dialog now grows a "Will bind" section under
the step-count badge. Each chord renders as a key-cap pill with the
chord string in mono, plus the when-predicate as a quieter
sub-label when set. When the chord conflicts with one already in
your local library, the pill border picks up the warn tone and a
`⚠ replaces an existing binding on "<title>"` line lands underneath.
Section is hidden when the workflow ships with no triggers, so
on-demand imports look exactly as they did in 1.0.

The actual rebind is unchanged. Hyprland and Sway pick up the new
KDL via the inotify watcher and swap the binding without a daemon
restart; portal-mode users restart the daemon to apply changes by
the GlobalShortcuts spec.

## What's still on the trigger backlog

- **KDE Plasma 6 + GNOME 46+ active-window probe** so portal users
  also get when-predicate gating. Vendor-specific D-Bus calls
  (`org.kde.KWin` / `org.gnome.Shell.Introspect`) that vary by
  version, so it's its own piece.
- **Hotstrings** still parse but don't fire. Different model than
  chord binding (continuous global keyboard monitor), waits for a
  later release.
- **Editing the chord on import** before accepting. The 1.0.1
  dialog surfaces the chord but doesn't let you remap. The right
  shape is probably "import first, open the Triggers tab on the new
  row" rather than a chord picker in the dialog itself.

## Upgrading

Drop-in. No state.toml or KDL changes; the new wrapper subcommand
is an additive CLI surface and `wflow run` keeps its existing
contract. AUR users get it via `pacman -Syu wflow`; tarball users
grab the new release.
