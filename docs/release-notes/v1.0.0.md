# wflow 1.0.0: the catalog goes live

The Explore tab is real. There's a public catalog of workflows at
[wflows.io](https://wflows.io), the desktop signs in to it, you can
import community workflows in one click, and you can publish your own
straight from the library. This is the release where wflow stops
being a personal automation tool you use alone and starts being a
shared library you build on with other people.

Pair it with the trigger daemon that landed in 0.7 and you have the
full shape: someone publishes a workflow on wflows.io, you import it
with one click, you bind it to ctrl+alt+t, and your compositor fires
it the next time you press the chord. Nothing about that loop existed
six weeks ago.

## Explore: a real catalog, not a mock

The Explore tab was hidden through 0.4 to 0.7 because it only had
fixture data behind it. It's on by default now and pointed at the
live wflows.io API. Featured rows, browse with sort and search and
tag filters, the workflow detail drawer with real install counts,
comments, parsed steps from the actual KDL source. The mock fixtures
are gone.

The detail drawer renders the live response: real installCount,
commentCount, publishedAt, updatedAt. The step preview parses the
inline KDL through the same decoder the runner uses, so what you see
on the drawer is what the engine would actually run. Shell commands,
key chords, durations, the lot.

## One-click import via wflow://

Click the Open in wflow button on any wflows.io page and the desktop
catches the `wflow://import?source=...` URL, fetches the workflow,
and shows a confirm dialog with title, author, description, and step
count before writing anything to disk. Cancel and nothing happens.
Confirm and the workflow lands in your library with a fresh id, ready
to run.

The dialog is the security beat. A drive-by page that triggers the
URL scheme can't silently install a workflow you didn't intend to
keep. You see what you're getting before it touches your config
directory.

The desktop is single-instance: a second `wflow` invocation forwards
its argv to the running process via a Unix socket and exits, so
deeplinks fire in the window you already have open instead of
spawning a new one.

## Sign-in, in your browser

Settings has an Account section now. Click Sign in and the desktop
opens your browser to wflows.io's sign-in flow. Approve, the page
redirects to a `wflow://auth?token=...` URL, the desktop catches it
and you're signed in. No webview embedded in the app. The token
persists in `state.toml` and survives restarts.

A nav-bar pill shows your handle (with a small green dot) when you're
signed in, "Sign in" when you're not, and "Signing in..." while the
browser handoff is in flight. It's the discovery point for everyone
who's never thought about the catalog before.

Once signed in, the Explore tab grows a Favorites row populated from
your account, and 401s on any authenticated call route through to a
clean sign-out so the UI doesn't leave you staring at stale data.

## Publishing, from the card

Right-click any workflow in your library when you're signed in and
the menu has a Publish to wflows.io item. The card itself grows a
small Publish pill in the top-right corner so the affordance is
visible without hunting. Click it and a dialog walks you through
description, tags, and visibility (public or draft). Submit and the
workflow posts to `/api/v0/workflows`; on success you get the public
URL, ready to share.

The KDL on disk is the source of truth. The publish flow reads your
local file, encodes it to KDL with the same encoder the CLI uses, and
posts it. Nothing converts through JSON in the middle. Whatever you
authored is exactly what other people install.

## The Triggers tab in the GUI

0.7 shipped the daemon but left chord authoring to hand-editing KDL.
1.0 has a Triggers tab that lists every active binding across your
library, lets you add or edit or remove them with a chord-capture
dialog, and writes the change back to the workflow's `trigger { }`
block. The daemon's file watcher picks it up on save.

Manual chord text entry is in there too, for the chords your
compositor is already grabbing globally before the daemon can catch
them. When-predicates (window-class and window-title) round out the
shape: bind ctrl+t to one workflow when Firefox is foreground and a
different one when it isn't.

The daemon also auto-enables itself on first GUI launch (one-time,
respects your decision afterwards) so a fresh install doesn't leave
you wondering why your chords don't fire.

## What changed under the hood

The brand domain is wflows.io across the codebase, the marketing
site, the site repo. The desktop's `WFLOW_SITE_ORIGIN` default points
there. The wflows.com domain is parked.

The KDL tokenizer on the website was tightened to accept the v2
bareword set the desktop emits (chord syntax with `+`, bare strings
in notify titles), so workflows you publish round-trip without "we
couldn't parse this KDL" errors on the receiving end.

About → Settings now reads the version from `env!("CARGO_PKG_VERSION")`
instead of a hardcoded string. One less spot to forget when cutting
releases.

## Upgrading

Drop-in. Workflows on disk round-trip unchanged. State.toml grows new
optional fields (`auth`, `daemon_unit_offered`) that default to the
right thing on existing installs. AUR users get it via
`pacman -Syu wflow`; tarball users grab the new release; Flatpak ETA
lines up with the next Flathub publish window.

If you were running the daemon under systemd from 0.7, no changes
needed. The user unit shipped in 0.7 is unchanged.

## What's not in 1.0

A few items that came up during the wflows.io build are explicitly
post-launch:

- **Auto-update notifier.** When a workflow you imported publishes a
  new version, the desktop should ping you. Schema's already there
  (`workflow_versions`, `remixedFromId`), the desktop side isn't.
- **OS hotkey trigger sync on import.** Workflows on wflows.io can
  carry a suggested chord; today the desktop doesn't apply it on
  import. After 1.0 the import path will append the chord into the
  KDL `trigger { }` block and let the daemon's file watcher pick it
  up.
- **Cloud sync.** Pro-tier territory. Workflows + folder structure
  syncing across machines. The schema's `private` visibility is the
  wedge but the desktop side is unbuilt.
- **Run-history telemetry.** Also Pro. Daemon-side recorder posting
  run outcomes for a per-user dashboard. Opt-in, with a delete-my-
  history path.

## What's next

With 1.0 out the AHK-on-Linux launch posts can finally drop. The
gating piece was always "is the daemon real and shipped" plus "does
the catalog have something on it"; both are true now.

Beyond that the backlog turns to polish (offline indicator on
Explore, fetch_browse caching, Settings advanced disclosure for motion
durations and origin override) and Pro-tier work (sync, run history,
auto-update). Roast whatever you find rough. The issues page is
[github.com/cushycush/wflow/issues](https://github.com/cushycush/wflow/issues).
